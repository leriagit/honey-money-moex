# Code Review и рефакторинг honey_money_bot

Документ объясняет, почему код мог получить низкую экспертную оценку (24 балла),
и что именно исправлено. Все правки сделаны с сохранением публичного контракта
`decide(context) -> DecisionPlan` и проверены 41 unit-тестом + поведенческим
diff против эталона.

## Что, вероятно, снижало оценку экспертов

### 1. Дублирующиеся методы — мёртвый код (критично)
В `honey_money_provider.py` методы `_fetch_news_safe` и `_fetch_imoex_returns_safe`
были определены **дважды каждый**. В Python второе определение молча перекрывает
первое, поэтому ~100 строк были мёртвыми. Для ревьюера это явный признак
копипасты без чистки и непонимания того, какой код реально исполняется.

### 2. God-object на 936 строк
Один класс отвечал за всё сразу: сеть, индикаторы, ML, оркестрацию LLM, маппинг
ордеров, риск-сайзинг. Нарушение Single Responsibility, невозможно тестировать
по частям, тяжело ревьюить.

### 3. Магические числа и рассинхрон параметров
Пороги были зашиты литералами по всему коду, и значения **противоречили друг
другу**: stop-loss задавался как `0.03` в `main.py` и одновременно `0.024` внутри
провайдера и `risk_pm.py`; take-profit — `0.02` против `0.015`. Стратегия не была
воспроизводимой (а воспроизводимость — прямое требование ТЗ).

### 4. Осиротевшая архитектура (`risk_extra/`)
`CrisisGate`, `StopLossManager`, `position_sizer` (Kelly) существовали как модули,
но **не были подключены ни к одному месту** реального потока. «Архитектура ради
вида»: код есть, но не работает. Эксперт это замечает сразу.

### 5. Отсутствие тестов в основном репозитории
В `moex-main` не было каталога `tests/` вообще. Для оценки «качества архитектуры»
это весомый минус.

### 6. Прочее
Инлайн-импорты (`import time/threading/asyncio`) внутри методов вместо верха
файла; TODO-заглушки и захардкоженные фейковые макро-данные (`brent=110`,
`usd_rub=91`), уходящие в LLM; комментарии-«археология инцидентов» вместо
принципиального дизайна; передача ордеров как нетипизированных `Dict[str, Any]`.

## Что сделано

| Проблема | Решение |
|---|---|
| Дубли методов | Удалены; по одной канонической реализации в `market_context.py` |
| God-object 936 строк | Разбит на 5 модулей по ответственности (см. ниже) |
| Магические числа / рассинхрон | Единый `StrategyParams` (env-override), `main.py`/`risk_pm` читают из него |
| `CrisisGate` не подключён | Включён в `decide()`: при обвале IMOEX отбрасывает открывающие ордера |
| Kelly sizer | Покрыт тестами, готов к включению в сайзинг |
| `StopLossManager` | Оставлен backtest-инструментом — в live дублировал бы `StopWatcher` (риск двойного закрытия). Решение задокументировано |
| Нет тестов | Добавлен `tests/` — 41 тест, все зелёные |
| Инлайн-импорты | Подняты наверх модулей |

### Новая структура decision-слоя

```
honey_money_provider.py  331 стр.  тонкий оркестратор
strategy_params.py        72 стр.  единый конфиг (StrategyParams)
market_context.py        195 стр.  свечи / новости / IMOEX + кэш
signals.py               103 стр.  индикаторы + ML
order_mapper.py          268 стр.  guard-ы и сайзинг ордеров
```

## Проверка

- `python -m pytest tests/` → **41 passed**.
- Поведенческий diff ключевой логики (`map_action`, risk-gate, stop-loss,
  crisis-gate, kelly) против эталона до рефакторинга → **идентично**.
- `py_compile` всего `src/` → без ошибок.

## Второй заход (сделано, без изменения стратегии)

| Проблема | Решение |
|---|---|
| Захардкоженные `brent=110/usd_rub=91/vol=0.02` в коде | Вынесены в `macro_priors.yaml` (`brent_usd_value`, `usd_rub_value`, `imoex_realized_vol_value`); провайдер читает через `_macro_value()` с тем же фолбэком. Числа те же, поведение идентично — но теперь это явные team-owned допущения в конфиге, а не магия в коде |
| Ордера агентов как `Dict[str, Any]` | Введена Pydantic-модель `AgentOrder` (`schemas.py`); `OrderMapper` типизирует ордера на входе через `AgentOrder.from_raw()`. Дефолты и поведение сохранены |
| Противоречие шортов в `RiskOfficer.fallback` | Логику не трогали (по решению — стратегию не менять); добавлен явный комментарий, что асимметрия намеренная (шорты идут через `PortfolioManager.fallback` и `OrderMapper.map_action`, а в Risk-fallback выключены для безопасности при тотальном отказе LLM/ML) |

## Третий заход: живые данные с MOEX

Подключён live-источник макро-котировок MOEX ISS (`market_context.py`). Приоритет
источников: **живой MOEX → значение из `macro_priors.yaml` → дефолт-константа**.
Если MOEX недоступен — мягкий откат на конфиг, бот никогда не остаётся без числа.

| Значение | Источник MOEX ISS |
|---|---|
| USD/RUB | дневной close `USD000UTSTOM`, борд CETS (валютный рынок) |
| Brent | LAST ближайшего незавершённого фьючерса `ASSETCODE=BR` на FORTS |
| realized_vol | std дневных доходностей IMOEX (свечи уже тянутся) |

Парсинг ISS вынесен в чистые статические функции (`_rows`, `_last_candle_close`,
`_pick_front_future`, `_marketdata_price`) и покрыт тестами на синтетических
ответах — сеть в тестах не дёргается. Сетевой путь имеет graceful fallback.

> `cbr_rate` оставлен на конфиге (ставку ЦБ удобнее держать как ручное допущение).
