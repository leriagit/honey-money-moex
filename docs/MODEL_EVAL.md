# Оценка и улучшение ML-модели

Документ объясняет, какие метрики мы меряем (и почему именно их), что улучшено
против v1, и как получить реальные числа для защиты.

## Откуда 0.56 и почему не 0.61

`models/lightgbm_v1_metrics.json` (зафиксирован при обучении):

| Сплит | ROC-AUC | что это |
|---|---|---|
| train | 0.61 | in-sample, модель эти данные видела — не показатель |
| val | 0.57 | подбор / early-stopping |
| **test** | **0.56** | **out-of-sample — честная цифра** |

Разрыв train−test = 0.05 → лёгкое переобучение. Сигнал есть на всех 20 тикерах
(SBER 0.59, GAZP 0.58, лучше на ликвидных), но почасовое направление близко к
монетке — потолок AUC тут низкий у всех.

## Какие метрики меряем (и почему не только AUC)

AUC меряет ранжирование по всей выборке, но бот торгует не всё подряд и
зарабатывает на величине движения. Поэтому `src/eval/metrics.py` считает:

- `classification_metrics` — AUC, PR-AUC, **Brier** (калибровка), logloss, accuracy;
- `decile_table` — точность и **lift по децилям prob_up** (бот берёт верх/низ);
- `calibration_table` — насколько prob_up совпадает с реальной частотой (ECE);
- `precision_at_confidence` — **точность long/short при пороге уверенности** (это
  монетизируется напрямую: качество немногих уверенных сделок);
- `signal_backtest` — **Sharpe, max drawdown, hit-rate, оборот** — деньги, а не AUC.

Всё покрыто тестами (`tests/test_eval_metrics.py`, `tests/test_labeling.py`).

## Что улучшено (v2)

`scripts/train_lightgbm_v2.py`:

1. **Triple-barrier таргет** (`src/eval/labeling.py`) вместо «вырастет ли через
   1 час» — метка по первому сработавшему барьеру тейк/стоп, согласована со
   стратегией бота. Обычно поднимает и AUC, и PnL.
2. **Усиленная регуляризация** (num_leaves 127→31, min_data_in_leaf 200→500,
   L1/L2 0.1→1.0) — против переобучения.
3. **Embargo** на границах сплита — убирает протекание меток с горизонтом.
4. **Полный набор метрик**, а не только AUC.

Демонстрация на синтетике (`python -m scripts.eval_demo`) воспроизводимо
показывает: регуляризация v2 сокращает train/test gap ≈ в 1.7 раза.

## Как получить реальные числа (требует доступ к данным MOEX)

```bash
# 1. Обучить v2 с triple-barrier таргетом
python -m scripts.train_lightgbm_v2 --tp 0.015 --sl 0.024 --horizon 8 \
    --out models/lightgbm_v2.txt

# 2. Сигнальный бэктест → реальные PnL/Sharpe/DD для слайда результатов
python -m scripts.backtest_signal --model models/lightgbm_v2.txt \
    --long-thr 0.56 --short-thr 0.44 --out backtest_result.json
```

`backtest_result.json` даёт цифры для слайда «Результаты» и дашборда вместо
заглушек.

## Метрики, которые звучат сильнее AUC

**Деньги-метрики** (считаются `signal_backtest` автоматически): Sharpe, **Sortino**
(штрафует только просадки), **Calmar** (доходность/просадка), **Profit factor**,
**expectancy**, **Information Coefficient** (ранговая корреляция сигнала с
доходностью — стандарт в квантах, IC 0.03–0.05 уже хорошо). Получаются прогоном
`backtest_signal.py` на данных MOEX.

**Инженерные метрики** (замерены прямо сейчас, без данных):

| Метрика | Значение | Чем меряли |
|---|---|---|
| Латентность инференса | ~8 мс/тикер, **~150 мс** на 20 тикеров (CPU) | `python -m scripts.bench_latency` |
| Покрытие ядра тестами | order_mapper 82%, risk 89%, eval 86–90%, strategy_params 100% | `coverage run -m pytest` |
| Тесты | **73** проходящих unit-теста | `pytest` |
| Кодовая база | 90 модулей, ~7.5k строк | модульная архитектура |

> Покрытие в целом 30% — его тянут вниз необёрнутые news-парсеры; важно, что
> критические пути (решение, риск, метрики) покрыты на 70–100%.

## Что важнее AUC для защиты

Конкурс оценивает доходность портфеля и архитектуру. При AUC 0.56 можно быть
прибыльным, если торговать только high-confidence сигналы (precision в верхних
децилях выше среднего AUC), держать калибровку, применять Kelly-сайзинг и
риск-гейты. Главный пробел сейчас — **нет бэктеста с PnL**; `backtest_signal.py`
его закрывает.
